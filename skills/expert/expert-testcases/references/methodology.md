# E2E测试用例集构建方法论（6步法）

## 方法论概述

本方法论从若依框架的实际业务模块（股权转让equity-transfer）中提炼，适用于所有含工作流的CRUD业务模块。

## 六步流程

### Step 1: 业务流分析

**目标**: 理解业务全貌，识别关键操作路径

**输入**: PRD文档 / 源码
**输出**: 业务流程图 + 操作清单

**方法**:
1. 从Controller识别业务操作（CRUD + 审批 + 导出）
2. 从Service识别业务规则（状态校验、文件校验、权限校验）
3. 从前端路由识别页面交互流程

### Step 2: 状态机建模

**目标**: 将业务流转化为可测试的状态转移模型

**输入**: Step 1输出
**输出**: 状态码集合 + 合法转移图

**方法**:
1. 提取`determineStatusByNode`方法中的节点→状态映射
2. 提取`handleApproval`中的approve/reject状态转移
3. 提取Controller中add/edit的初始状态设置
4. 构建状态转移矩阵

**若依框架状态码模式**:
```
00: 初始
01: 待提交（草稿）
02: 已提交/审批中
03~07: 各级审批通过
08: 已驳回
09: 已终止
```

### Step 3: API契约提取

**目标**: 自动化提取所有API端点及其元信息

**输入**: Controller源码
**输出**: API端点清单（method/path/permission/description）

**提取规则**:
| 注解 | 提取内容 |
|------|---------|
| @RequestMapping | 基路径 |
| @GetMapping | GET端点 |
| @PostMapping | POST端点 |
| @PutMapping | PUT端点 |
| @DeleteMapping | DELETE端点 |
| @PreAuthorize | 权限标识 |
| @Log | 操作日志标记 |

### Step 4: 角色权限矩阵

**目标**: 构建角色×操作的权限矩阵，确保权限测试覆盖

**输入**: @PreAuthorize注解 + 角色配置
**输出**: 权限矩阵表

**矩阵维度**:
- 行: 5个测试角色（applicant/dept_manager/leader/admin/cross_dept_user）
- 列: 所有API端点
- 值: ✅有权限 / ❌无权限 / ⚠️越权

### Step 5: 测试用例生成

**目标**: 基于前4步的分析结果，生成6层测试用例

**6层测试金字塔**:

| 层级 | 用例ID前缀 | 关注点 | 代码框架 |
|------|-----------|--------|---------|
| API层 | A | 单接口输入输出 | pytest + requests |
| 集成层 | I | 多接口联动+状态流转 | pytest + requests |
| UI层 | U | 页面交互+表单验证 | Playwright |
| E2E层 | E | 完整业务场景 | Playwright |
| 边界异常层 | B | 边界值+异常输入 | pytest |
| 性能层 | P | 并发+响应时间 | Locust |

**用例编写规范**:
每条用例必须包含8个字段：用例ID、测试目的、优先级、类型、前置条件、操作步骤、预期结果、状态

### Step 6: 数据驱动封装

**目标**: 将测试数据与测试逻辑分离，实现数据驱动

**输出**: test-data.yaml标准格式文件

**数据分类**:
1. 测试用户（5角色）
2. 表单数据变体（valid/invalid/boundary）
3. 审批操作数据
4. 文件上传数据
5. 工作流配置
6. 状态码定义
7. API端点清单

## 应用验证

本方法论已在ZCCZ-1股权转让模块中完整验证，产出：
- 97个文档用例 + 63个pytest可运行函数
- 覆盖13个API端点、10个状态码、5个角色
