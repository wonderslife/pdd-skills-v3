## 1. 假设提出模板

基于Bug定位信息，提出Top 3可能的根因假设：

```yaml
假设列表:
  - id: H1
    description: "Vue 2响应式系统无法检测深层嵌套数组变化"
    category: FRAMEWORK_LIMITATION
    evidence:
      - "+15": 常见问题模式（社区高频问题）
      - "-20": 未经验证（尚未添加console.log确认方法被调用）
    confidence: 25%  # 初始置信度
    
  - id: H2
    description: "事件绑定语法错误或事件未正确触发"
    category: EVENT_BINDING_ISSUE
    evidence:
      - "+20": 用户报告"点击无反应"
      - "+15": 控制台无任何输出"
    confidence: 65%
    
  - id: H3
    description: "存在同名方法导致方法被覆盖"
    category: METHOD_CONFLICT
    evidence:
      - "+20": 文件中有多个删除相关方法
      - "-10": 需要代码搜索确认
    confidence: 60%
```

## 2. 置信度评分详细规则

### 计算公式

```
置信度 = 基础分(30) + 证据加分(最多+35) - 扣分项(最多-50)
```

### 证据加分项（最多+35）

| 证据类型 | 分值 | 判定标准 | 示例 |
|---------|------|---------|------|
| 有用户明确报告 | +15 | 用户明确描述了关键现象 | "点击后控制台完全没反应" |
| 有代码层面证据 | +20 | 已发现可疑的代码模式 | 发现可疑的同名方法定义 |
| 是常见/高频问题模式 | +10 | 该问题在社区/项目中高频出现 | Vue响应式问题是高频问题 |
| 与症状完全匹配 | +15 | 假设能解释所有观察到的现象 | 假设涵盖了用户报告的所有症状 |

### 扣分项（最多-50）

| 扣分类型 | 分值 | 判定标准 | 示例 |
|---------|------|---------|------|
| 未经验证 | -20 | 没有最小化测试证明假设 | 没有console.log确认方法被调用 |
| 与已知事实矛盾 | -30 | 假设与已观察到的事实冲突 | 假设要求方法被调用但控制台无输出 |
| 过于复杂/牵强 | -15 | 假设涉及多层间接依赖 | 需要A→B→C→D四层推理链 |

### 评级标准与行动建议

| 置信度范围 | 评级 | 行动建议 | 说明 |
|---------|------|---------|------|
| ≥ 80% | ✅ **强确信** | 可直接作为首选方案pursue | 高度可信，可直接进入修复阶段 |
| 60%-79% | ✅ **较确信** | 应该优先验证此假设 | 较可信，应设计验证测试 |
| 40%-59% | ⚠️ **存疑** | 可以尝试验证，但不要排除其他可能 | 中等可信，作为备选验证 |
| < 40% | ❌ **弱假设** | 不应作为首选，仅作最后备选 | 低可信，不建议投入时间验证 |

## 3. 验证协议详细步骤

### Step 1: 设计最小化测试

创建最简单的测试用例来验证假设：
- **目标**: 用最少代码复现假设预测的现象
- **方法**: 添加一行 `console.log` 或 `alert`
- **原则**: 测试应该能在1分钟内完成

### Step 2: 执行测试并收集证据

请用户执行测试并提供反馈：
- 控制台是否有输出？
- 输出内容是什么？
- 是否符合假设的预测？

### Step 3: 对比预期与实际

| 实际结果 | 判定 | 后续行动 |
|---------|------|---------|
| ✅ 完全符合预测 | 假设成立 | 置信度提升至80%+，进入修复阶段 |
| ❌ 完全不符合 | 假设否定 | 置信度降至10%，立即转向其他假设 |
| ⚠️ 部分符合 | 需要细化 | 收集更多信息，拆分或修正假设 |

### Step 4: 迭代或转向

```
如果Top 3假设全部被否定 → 回到Step 1.1重新收集信息
如果某个假设得到验证 → 进入Step 2 影响分析
如果部分符合但不确定 → 提出更细粒度的子假设
```

## 4. 验证示例库

### 示例1: Vue响应式问题假设（来自真实案例）

**假设**: H1 - "Vue 2无法检测深层嵌套数组变化"  
**初始置信度**: 25%

**验证设计**:
```javascript
// 在目标方法第一行添加
removeResolutionFile(decisionIndex, fileIndex) {
  console.log('=== METHOD CALLED ===', arguments); // 最小化测试
  console.log('form:', this.form.internalDecisions);
  
  // 原有代码...
}
```

**执行步骤**:
1. 请用户点击删除按钮
2. 观察浏览器控制台输出

**预期结果（如果假设成立）**:
- ✅ 控制台显示 `METHOD CALLED` + 参数信息
- ✅ 但数组splice操作后视图不更新

**实际结果**:
- ❌ 控制台完全没有任何输出
- 这意味着**方法根本未被调用**

**结论**:
- 假设H1被否定（置信度: 25% → 10%）
- 真正的问题在更上游：事件绑定或方法存在性
- 下一步: 转向假设H2或H3进行验证

---

### 示例2: 方法名冲突假设（来自真实案例）

**假设**: H3 - "存在同名方法导致方法被覆盖"  
**初始置信度**: 60%

**验证设计**:
```bash
# 执行代码搜索
grep -n "removeResolutionFile" detail.vue
```

**预期结果（如果假设成立）**:
```
Line 823:   removeResolutionFile(decisionIndex, fileIndex) {  // 定义点A
Line 1066:  removeResolutionFile(index) {                      // 定义点B ⚠️ 冲突！
Line 316:   @click="removeResolutionFile(index, scope.$index)" // 调用点
```

**实际结果**: 符合预期，发现两个同名方法

**结论**:
- 假设H3得到验证（置信度: 60% → 85%）
- 根因确定：JavaScript后定义的方法覆盖了先前的定义
- 下一步: 进入修复阶段（重命名冲突方法）

---

### 示例3: 事件绑定问题假设

**假设**: H2 - "事件绑定语法错误导致事件未触发"  
**初始置信度**: 65%

**验证设计**:
```html
<!-- 修改前 -->
<el-button @click="removeResolutionFile(index, fileIndex)">删除</el-button>

<!-- 临时修改为（添加最简日志）-->
<el-button @click="
  console.log('=== EVENT FIRED ===', index, fileIndex);
  removeResolutionFile(index, fileIndex);
">删除</el-button>
```

**判定逻辑**:

| 控制台输出 | 含义 | 结论 |
|-----------|------|------|
| 显示 `EVENT FIRED` | 事件已触发 | ✅ 事件绑定正常，问题在后续环节 |
| 无任何输出 | 事件未触发 | ❌ 事件绑定有问题，检查语法/修饰符 |

## 5. 常见错误模式与纠正

### 错误1: 跳过验证直接修复

❌ **错误做法**:
```
假设: "这看起来像Vue响应式问题"
直接实施: 使用$forceUpdate()强制渲染
结果: 失败，浪费30分钟
```

✅ **正确做法**:
```
假设: "这看起来像Vue响应式问题" (置信度: 25%)
验证: 添加console.log确认方法是否被调用
发现: 控制台无输出 → 假设错误
转向: 检查事件绑定或方法存在性
```

### 错误2: 验证不充分就下结论

❌ **错误做法**:
```
验证: 添加了一行日志但未要求用户反馈
假设: "应该没问题了"
结果: 基于猜测继续，仍然失败
```

✅ **正确做法**:
```
验证: 添加日志 + 明确请求用户执行+反馈
等待: 用户报告控制台输出
分析: 基于实际证据判断假设成立与否
决策: 根据证据调整方向
```

### 错误3: 固守单一假设

❌ **错误做法**:
```
坚持假设H1 (Vue响应式问题)
尝试了7种不同的修复方式都失败
拒绝考虑其他可能性
总耗时: 2小时
```

✅ **正确做法**:
```
假设H1失败后 (第1次尝试)
立即转向假设H2 (事件绑定)
H2也失败后 (第2次尝试)
转向假设H3 (方法名冲突)
H3成功 (第3次尝试)
总耗时: 30分钟
```
